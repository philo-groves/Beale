# Source Index: Vulnerability Research Agent Harnesses

Research date: 2026-04-26.

This index prioritizes primary sources from AI labs, AI safety institutes, security research groups, benchmark creators, and major institutions.

## High-Signal Sources

| Source | Organization | What it contributes | Harness implication for Beale |
| --- | --- | --- | --- |
| CyberGym project and paper | UC Berkeley | Large-scale benchmark with 1,507 real-world vulnerabilities across 188 projects. Level 1 gives vulnerability descriptions and unpatched codebases; success is a working PoC that triggers pre-patch and not post-patch. The project also reports incomplete patches and zero-days from generated PoCs. | Beale should support pre/post-patch verification, PoC artifacts, latest-version replay, and open-ended discovery modes. Source: https://www.cybergym.io/ and https://arxiv.org/abs/2506.02548 |
| CyberGym leaderboard JSON | UC Berkeley | Verified the user's example: `GPT-5.4` with `Codex CLI` is listed at 0.663 and `GPT-5.4` with `OpenAI Agent` is listed at 0.790 for one-trial Level 1. That is a 12.7 percentage point gain, rounded to 13 points, from harness/scaffold change with the same model. | The harness must be evaluated independently from model choice. Beale should record model, agent scaffold, prompt version, tool set, attempts, token budget, and environment version as first-class run dimensions. Source: https://www.cybergym.io/assets/results/leaderboard.json |
| Project Naptime | Google Project Zero | Defines five practical principles: space for reasoning, interactive environment, specialized tools, perfect verification, and sampling strategy. Reports up to 20x improvement over the original CyberSecEval2 methodology by changing evaluation/harness design. | Beale should avoid single-shot answers. It needs an iterative action-observation loop with code browsing, scripting, debugger/runtime tools, verifier feedback, and independent trajectory sampling. Source: https://projectzero.google/2024/06/project-naptime.html |
| From Naptime to Big Sleep | Google Project Zero and DeepMind | Shows variant analysis on real code. The agent used commit messages/diffs as a research seed, explored code, adapted failed tests, built a reproducer, and produced root-cause analysis. | Add task mode for variant analysis seeded by a patch/diff. Store failed hypotheses and environment feedback because they are part of the discovery path. Source: https://projectzero.google/2024/10/from-naptime-to-big-sleep.html |
| EVMbench | OpenAI, Paradigm, OtterSec | Evaluates detect, patch, and exploit modes for smart contracts. Uses deterministic local blockchain replay, isolated grading containers, restricted RPC methods, oracle patches/exploits, and red-teamed graders. | Beale should define separate modes for detection, patching, and reproduction/exploitation. The grader must be isolated from the agent and should be red-team reviewed for cheating paths. Source: https://openai.com/index/introducing-evmbench/ and https://cdn.openai.com/evmbench/evmbench.pdf |
| Codex Security research preview | OpenAI | Describes project-specific threat modeling, validated findings, sandboxed validation, and patches generated with full system context. Reports major reductions in noise and severity over-reporting during beta. | Beale should maintain an editable threat model per target and make validation evidence central to every finding. Findings without reproduction or high-confidence evidence should stay in triage. Source: https://openai.com/index/codex-security-now-in-research-preview/ |
| GPT-5.3-Codex system card cyber section | OpenAI | Uses three cyber eval classes: professional CTFs, CVE-Bench, and internal Cyber Range. Notes limitations of CTFs and web-only CVE benchmarks. Uses many rollouts and headless Linux tooling. | Beale should not overfit to CTF-style tasks. It needs long-horizon cyber-range style workflows, consistency metrics, and scenario-level evaluation. Source: https://deploymentsafety.openai.com/gpt-5-3-codex/cybersecurity |
| OpenAI Agents/Responses docs | OpenAI | Responses API is agentic by default with tool use, stateful context, and built-in tools. Agents SDK supplies agent loops, handoffs, guardrails, sessions, and tracing. OpenAI's sandbox-agent cookbook recommends keeping the trusted harness outside the sandbox while using the sandbox as a tool. | Beale should treat the host GUI/harness as the trusted controller and target execution as disposable sandboxes. Built-in tracing and handoffs are likely useful, but benchmark control may require custom orchestration over the Responses API or Agents SDK. Sources: https://developers.openai.com/tracks/building-agents, https://developers.openai.com/api/docs/guides/migrate-to-responses, https://developers.openai.com/cookbook/examples/agents_sdk/sandboxed-code-migration/sandboxed_code_migration_agent |
| Project Glasswing / Claude Mythos Preview | Anthropic | Reports frontier model vulnerability discovery and exploitation capability, with restricted defensive access for major software/security organizations. Publishes CyberGym result of 83.1 percent for Mythos Preview vs 66.6 percent for Opus 4.6. | The practical future is assisted defensive vulnerability discovery at scale. Beale should include guardrails, audit logs, authorization boundaries, and maintainer-oriented reporting from the start. Source: https://www.anthropic.com/glasswing |
| BountyBench | Stanford AI Lab and collaborators | Benchmarks real systems across Detect, Exploit, and Patch. Uses code/runtime invariants, manually reproduced exploits, verified patches, bug bounty dollar values, and difficulty modulation by information provided. | Beale should measure both offensive and defensive outcomes, include runtime health checks, and score findings by impact/evidence rather than raw count. Source: https://ai.stanford.edu/blog/bountybench/ and https://arxiv.org/abs/2505.15216 |
| AISI international agentic testing exercise | UK AISI, EU AI Office, Australia, others | Evaluated how temperature, attempts, token limits, prompts, and tools affect agentic cyber performance. Recommends quick sweeps, token limits beyond diminishing returns, and examining environments/transcripts for task bugs. | Beale should have a built-in harness optimizer that runs small calibration sweeps before large evals. It should flag task/environment bugs separately from model failures. Source: https://www.aisi.gov.uk/work/international-joint-testing-exercise-agentic-testing |
| US/UK AISI pre-deployment evaluation of OpenAI o1 | US AISI and UK AISI | Notes that early o1 had tool-calling and output-format issues that evaluators mitigated through prompt and agent design changes. Cyber tasks used task-specific Kali containers, command timeouts, Python state, multiple attempts, and SME transcript review. | Beale should include model-specific adapter layers, robust tool-call recovery, and expert review surfaces for transcripts. Source: https://www.nist.gov/system/files/documents/2024/12/18/US_UK_AI%20Safety%20Institute_%20December_Publication-OpenAIo1.pdf |
| EnIGMA | Princeton, NYU, Tel Aviv, Stanford, others | Adds Interactive Agent Tools for debuggers and server connections. Reports strong gains on CTF benchmarks and calls out data leakage plus "soliloquizing", where the model invents observations instead of interacting with the environment. | Beale should design interactive tools as structured tools, not raw terminal-only text. The trace system must distinguish actual observations from model claims. Source: https://arxiv.org/abs/2409.16165 |
| NYU CTF Bench | NYU and collaborators | Open benchmark and automated framework for offensive-security CTF challenges across categories, designed for interactive cybersecurity task planning. | Useful for broad tool coverage, but should not be the only target because CTFs are narrower than real vulnerability research. Source: https://openreview.net/forum?id=itBDglVylS |
| Cybench | Stanford / Epoch AI | Measures end-to-end agent performance on sandboxed security tasks requiring reconnaissance, tool use, multi-step planning, and persistence; reports unguided and subtask-guided settings. | Beale should track both unguided and guided performance, plus time-to-solve and partial progress. Source: https://arxiv.org/abs/2408.08926 and https://epoch.ai/benchmarks/cybench |
| CyberSecEval 3 | Meta | Expands LLM cybersecurity risk/capability evaluation to offensive security, scaling manual operations, autonomous offensive operations, social engineering, and mitigations. | Useful taxonomy for risk categories, but Beale's core harness should focus on authorized vulnerability research and defensive validation. Source: https://arxiv.org/abs/2408.01605 |
| CVE-Bench and one-day vulnerability studies | Academic cyber-agent research | CVE-Bench targets real-world web app vulnerabilities in sandboxed scenarios. Earlier one-day work shows large performance differences when a CVE description is provided versus not provided. | Beale should explicitly encode information regimes: no-info discovery, weakness hint, patch/diff hint, vulnerability report, and full reproduction target. Sources: https://arxiv.org/abs/2503.17332 and https://arxiv.org/abs/2404.08144 |

## Immediate Takeaway

The strongest shared pattern is not a single agent architecture. It is a benchmark-grade environment that lets the model iterate, inspect, execute, validate, retry independently, and leave a complete audit trail.
